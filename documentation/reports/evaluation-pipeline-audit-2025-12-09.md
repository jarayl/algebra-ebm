## Executive Summary

The evaluation pipeline has **THREE CRITICAL BUGS** causing systematic failure during decoding and inference:

1. **BUG #1 (CRITICAL):** Decoder candidate set mismatch - evaluation uses ~49 hardcoded equations while test datasets generate thousands of unique equations
2. **BUG #2 (CRITICAL):** Distance threshold chaos - thresholds vary wildly (1.0→6.0→50.0) across different code paths as emergency workarounds
3. **BUG #3 (SEVERE):** Evaluation script bypasses the decoder rebuild logic in algebra_evaluation.py by creating decoder with wrong threshold upfront

**Root Cause:** The decoder cannot find matches because it's searching in the wrong candidate set with the wrong distance metric.

**Impact:** Near-zero evaluation accuracy despite correct model training.

**Confidence:** HIGH - Multiple independent code paths confirm the same issues.

### Finding 4: Encoder Normalization Not Accounted For

**Evidence:**

**Location:** `algebra_encoder.py:131-136`
```python
# CRITICAL FIX: Normalize embeddings for diffusion process
# The diffusion model expects data roughly in [-1, 1] range
# Normalizing to unit L2 norm ensures bounded, well-behaved embeddings
if self.normalize_embeddings:
    embedding = torch.nn.functional.normalize(embedding, p=2, dim=-1)
```

**Analysis:**
1. All embeddings are normalized to unit L2 norm (||e|| = 1)
2. For normalized embeddings:
   - Maximum possible distance = √2 ≈ 1.414 (opposite points on unit sphere)
   - Typical same-equation distance ≈ 0.0
   - Typical similar-equation distance ≈ 0.2-0.5
   - Different-equation distance ≈ 0.8-1.2
3. Original threshold=1.0 was reasonable for this
4. Threshold=50.0 accepts EVERYTHING (distance can never exceed ~1.4)

**Impact:** With threshold=50.0 and normalized embeddings, the decoder will match any random embedding to SOME candidate, even if completely wrong.

**Confidence:** HIGH - Mathematical analysis + code evidence

**Evidence:**

**Location:** `algebra_inference.py:746-757`

**Location:** `algebra_encoder.py:558-562`

**Problem:** Double threshold checking:

**Impact:** Not a direct bug, but confusing control flow. The real issue is that with threshold=50.0, the decoder ALWAYS returns a match (never None).

**Confidence:** MEDIUM - Code structure issue, not correctness bug

## Appendix A: Distance Threshold Math

For normalized embeddings where ||e|| = 1:

```
Maximum distance = ||e1 - e2|| where e1 and e2 are antipodal
                 = ||e1||² + ||e2||² - 2(e1·e2)
                 = 1 + 1 - 2(-1)
                 = sqrt(4) = 2.0   [ERROR IN EARLIER ANALYSIS]

Wait, let me recalculate:
||e1 - e2||² = (e1-e2)·(e1-e2)
             = e1·e1 - 2*e1·e2 + e2·e2
             = ||e1||² + ||e2||² - 2*e1·e2
             = 1 + 1 - 2*e1·e2
             = 2(1 - e1·e2)

Maximum when e1·e2 = -1 (opposite):
  ||e1-e2||² = 2(1-(-1)) = 4
  ||e1-e2|| = 2.0

So maximum distance is 2.0, not 1.414!
```

This means:
- threshold=1.0 is REASONABLE (half of max)
- threshold=2.0 is SAFE (at max distance)
- threshold=6.0 is EXCESSIVE (3x max possible)
- threshold=50.0 is ABSURD (25x max possible)
